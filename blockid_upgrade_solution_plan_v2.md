# 1. Executive Decision

BlockID should not try to beat Cake Equity, Carta, Pulley, Ledgy, or Orchestra as a pure cap table platform in the first 6 months.

The sharper wedge is:

> **Australia-native fundraising readiness + cap table / term sheet hygiene score for founders raising now.**

BlockID should sell first as:

> **AI Fundraising Readiness + Term Sheet / Dilution Copilot for Australian founders.**

The larger product direction should now be framed as:

> **AI trust layer and enterprise platform for startup fundraising, valuation and ownership management.**

The biggest product differentiator is:

> **AI-powered fundraising, valuation and ownership infrastructure for private companies.**

BlockID can be packaged as a SaaS platform or standalone/private deployment for SMEs, startups, accelerators, advisors, investors, shareholders and related private-company stakeholders. AI is an intelligence layer inside the product, not the primary brand positioning.

Blockchain remains important, but its first role is not a chain, token, marketplace, or liquidity product. Its first role is **proof infrastructure**:

- verified score snapshots
- document integrity proofs
- cap table provenance
- governance attestations
- data room access audit trails

This keeps the product commercially useful, legally safer, and easier to sell.

# 4. Competitor Learnings

## Direct Equity / Cap Table Platforms

**Cake Equity** is the most dangerous local competitor because it is Australia-native and founder-friendly. It covers cap table, ESOP, capital raise workflows, contracts, employee app, bare trusts, and global pools.

**Carta, Pulley, Ledgy, Eqvista, Mantle, and Orchestra** compete through depth: cap table operations, equity plans, compliance, stakeholder portals, document management, valuations, and scenario modeling.

## Legal / Fundraising Workflow Platforms

**Lawpath, Sprintlaw, SeedLegals, and Clerky** compete for startup legal budget. They do not own the full investor readiness workflow, but they are close to the point where founders feel pain.

## Investor CRM / Data Room / Readiness Platforms

**Visible, OpenVC, VCReady, Pitchago, and similar products** compete around investor updates, pitch readiness, fundraising CRM, data room analytics, and readiness scoring.

## Gap BlockID Can Own

No obvious player owns the complete Australia-native pre-diligence layer:

- ASIC / ABR identity
- cap table hygiene
- term sheet risk
- dilution impact
- data room readiness
- investor share link
- AU compliance signals
- tamper-evident proof trail

This is BlockID's near-term category:

> **The pre-diligence trust layer for Australian startup fundraising.**

# 8. Blockchain / Proof Upgrade Plan

## Principle

Off-chain systems are the operational source of truth. Blockchain or public timestamping is used only to prove integrity and timing.

Use blockchain when:

- multiple parties do not fully trust each other
- a document or score must be proven unchanged
- BlockID should not be able to quietly rewrite history

Do not use blockchain to store:

- PDFs
- personal identity documents
- passport / KYC data
- financial statements
- raw cap table data
- personal information that may need deletion or correction

## P0 — Verified Score Snapshot

Every Investor-Ready Score should produce a proof record.

Hash:

- normalized inputs
- score result
- score version
- model / rules version
- timestamp
- company id
- PDF hash

Add:

- proof id
- verify page
- downloadable proof certificate

Sales message:

> Every Investor-Ready Score is tamper-evident.

## P1 — Document Proof Vault

Allow founders to create integrity proofs for:

- term sheets
- SAFEs
- share certificates
- shareholder agreements
- board resolutions
- investor-ready PDFs

Store:

- document hash
- document type
- company id
- uploader / signer reference
- timestamp
- proof status

Do not claim this replaces legal execution.

## P1 — Cap Table Provenance

Represent cap table changes as events:

- `share_issued`
- `safe_signed`
- `safe_converted`
- `convertible_note_issued`
- `esop_granted`
- `option_exercised`
- `transfer_recorded`
- `round_closed`
- `certificate_voided`

Each event links to:

- actor
- approval
- document proof
- timestamp
- previous state hash
- new state hash

## P2 — Governance Attestation

Attest:

- board approval
- shareholder approval
- ESOP top-up approval
- round approval
- share issue approval

Each attestation should include:

- resolution id
- company id
- approval type
- quorum status
- signer references
- document hash
- timestamp

## Recommended Stack

First 90 days:

- Supabase / Postgres append-only event log
- canonical JSON serialization
- SHA-256 or BLAKE3 hashes
- Merkle tree batch roots
- internal verification page

After validation:

- OpenTimestamps, low-cost EVM L2, or simple anchoring contract
- W3C Verifiable Credentials for investor / company status
- `did:web:blockid.au` for BlockID issuer identity

Do not start with Cosmos SDK or sovereign chains in the first 90 days.
